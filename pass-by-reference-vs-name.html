Pass-by-Reference
vs Pass-by-Name

Understanding Fixed Memory Binding vs Re-Evaluated Expressions

🔗
Pass-by-Reference

Fixed Binding

Binds to a single memory address at call time. All subsequent accesses hit the same location, even if the index expression changes.

🔄
Pass-by-Name

Re-evaluated Each Use

The expression is re-computed every time the parameter is accessed. Changes to variables inside the expression affect future evaluations.

Expression Parameter Passing

Why Pass-by-Name is unique — expressions without a memory address

🔗 Pass-by-Reference

i = 1
P(i + 1)

procedure P(x)
begin
print(x)
i = 5
print(x)
end
❌ Cannot pass i + 1 by reference — expressions have no memory address. A reference parameter requires a variable or l-value.

🔄 Pass-by-Name

i = 1
P(i + 1)

procedure P(x)
begin
print(x)
i = 5
print(x)
end
First evaluation i + 1 = 1 + 1 = 2
Then: i = 5 Variable i changes to 5
Second evaluation i + 1 = 5 + 1 = 6
Output: 2 6

What Does "Re-evaluated Each Use" Actually Mean?

The fundamental difference in how each strategy resolves parameter access

🔗 Pass-by-Reference

The binding is determined once at call time. The parameter becomes an alias for a specific memory location.

↓
🔒 Address Chosen Once
↓

↓
📋 Expression Saved
↓

The Swap Problem

The classic example that demonstrates why Pass-by-Name can be dangerous

Pass-by-Reference

Works as expected — addresses are fixed

i = 1
A[1] = 10
A[2] = 20

swap(i, A[i])
Call: swap(i, A[i]) x → address of i (fixed)
Call: swap(i, A[i]) y → address of A[1] (fixed)
Inside: temp = y; y = x; x = temp A[1] gets i's value, i gets A[1]'s value
Result: i = 10, A[1] = 1, A[2] = 20
✅ Works as expected. Both parameters refer to fixed memory addresses regardless of what happens inside the procedure.

Pass-by-Name

Surprising behavior — indices change mid-execution

i = 1
A[1] = 10
A[2] = 20

swap(i, A[i])
Call: swap(i, A[i]) x → i (re-evaluates)
Call: swap(i, A[i]) y → A[i] = A[1] = 10
Step: temp = y temp = A[1] = 10
Step: y = x A[i] = i → A[1] = 1
Step: x = temp i = temp = 10
But wait — i is now 10! y = x was A[1] = 1 (correct)
But x = temp means i = 10
Final state i = 10, A[1] = 1, A[2] = 20
Result: i = 10, A[1] = 1, A[2] = 20
⚠️ Actually works here — but because A[i] is re-evaluated, if the array were larger and A[i] changed to point to a different element mid-swap, the result would be completely wrong. This unpredictability made Pass-by-Name notorious.

Summary

The key takeaway — choose your binding strategy wisely

🔗

Pass-by-Reference

"Choose Address Once"
  • ✅ Fixed binding — predictable
  • ✅ Memory address — efficient
  • ✅ Fast — no re-evaluation
  • ❌ Cannot accept expressions
  • ❌ Less flexible
🔄

Pass-by-Name

"Choose Meaning Every Time"
  • ✅ Dynamic binding — flexible
  • ✅ Expression evaluation — can pass any calculation
  • ✅ Flexible — adapts to context
  • ❌ Complex — hard to reason about
  • ❌ Unpredictable — surprising side effects
